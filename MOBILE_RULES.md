# Mobile-dev rules

Правила для разработки с телефона через `claude-telegram-bot` или `claude` CLI. Когда Артём работает не за компьютером, эти правила минимизируют шанс сломать прод и потерять контекст.

## 1. ОДИН АГЕНТ НА ПРОЕКТ
Не запускай вторую сессию правок в том же репо пока первая не закоммитила. Параллельные правки одного файла = merge-конфликт = потерянные часы. Если есть `bybit-claude-watchdog` или `bybit-claude-hourly` — это автономные процессы со своим контекстом, не вмешивайся пока они работают.

## 2. PLAN MODE ДЛЯ ИЗМЕНЕНИЙ >3 ФАЙЛОВ
Если задача требует трогать больше 3 файлов или вводит новую фичу — **сначала плана**. Текстом или голосом: «План: 1) X, 2) Y, 3) Z. Файлы: A, B, C. Тесты: D». Жди «да» от Артёма. Только потом исполняй.

## 3. SMALL SCOPE = ONE THING PER SESSION
Один баг ИЛИ одна feature за сессию. «Поправить тайминг VAD» — да. «Зарефакторить voice-tutor под FastAPI» — нет, это с компа. Если задача оказалась больше чем выглядела — **скажи об этом и предложи разбить**, не лезь дальше.

## 4. VOICE FOR INTENT, TEXT FOR NAMES
Голос неточен для имён файлов/функций (Whisper глушит русские буквы, сливает английские термины). Правило:
- **Голосом**: что хочешь сделать («поправь баг с памятью в трейдинг-боте»)
- **Текстом / скриншотом**: имена функций, пути, команды, логи (`process_signal`, `/root/.../config.json`)

Если получил голосовой запрос с именами файлов — переспроси текстом.

## 5. COMMIT + PUSH AS CHECKPOINT
После каждого работающего фикса (тесты прошли / сервис рестартанул чисто) — `git add` + `git commit` + **`git push`**. Артём 2026-05-02 выбрал auto-push: он не хочет пушить руками после каждой Claude-сессии. Push идёт сразу за commit'ом по умолчанию.

Исключения когда НЕ пушить:
- Локальные эксперименты в feature-ветке которая ещё не готова — спроси Артёма
- Если push fails (auth/network) — не повторяй слепо в цикле, докладывай ошибку
- `git push --force` к main/master — НИКОГДА без явного «force push, я знаю что делаю» от Артёма

Если push оказался ошибочным (плохой commit) — `git revert` + commit + push (forward fix). НЕ `git reset --hard` + `push --force`.

## 6. READ MEMORY FIRST
При первом взаимодействии в сессии (или при `/clear` + новая тема):
1. Прочитай `<project>/CLAUDE.md` — структура и ограничения проекта
2. Прочитай релевантные `/root/.claude/projects/-root/memory/project_*.md` и `feedback_*.md`
3. Только после — отвечай. Никогда не действуй без памяти.

## 7. REPORT EVERY DESTRUCTIVE OP
Перед edit, restart, commit, удалением, изменением конфига — **одна короткая строка**: «делаю X потому что Y». Потом исполняй. После — подтверждение. Никаких молчаливых правок.

Деструктивные операции: `rm`, `git reset --hard`, `git push --force`, `systemctl stop/disable`, `DROP TABLE`, изменение `.env`, `.service` файлов, миграция БД.

## 8. STATE FIRST: ALWAYS CHECK BEFORE TRUST
Память может быть устаревшей. Перед действием на основе памяти:
- Имена файлов/функций — `grep` или `ls`
- Состояние сервиса — `systemctl status <name>`
- Состояние git — `git status`, `git log -3`
- DB-состояние — `sqlite3 ... '.schema'` или `psql ... '\d'`

Если память расходится с realитой — обнови память, не действуй на сломанной модели.

## 9. STOP AT AMBIGUITY
Если задача неоднозначна и есть >1 разумная интерпретация — **не выбирай молча**. Спроси: «Я понял X, или ты имел в виду Y?». Один тап Артёма дешевле часа неправильной работы.

## 10. NO SECRETS COMMITTED
`.env`, `.credentials.json`, ключи, токены — **никогда** в git. Если случайно — `git rm --cached`, ротация ключа, новый коммит.

## 11. MEMORY SYNC ON COMMIT
После push в основной репо — синхронизируй `/root/.claude/projects/-root/memory/` в `Claude-AI-md-files`. Это резервная копия памяти.

## Когда mobile-разработка работает плохо
Если ты замечаешь любое из этого — **скажи Артёму что эту задачу лучше делать с компа**:
- >10 файлов в одном изменении
- Архитектурный outline / новая система с нуля
- Дебаг падающего prod где надо смотреть несколько логов параллельно
- Конфликтный git merge
- Длинные multi-step миграции БД
